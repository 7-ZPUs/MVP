# Context and Objective

We are building a Node.js/TypeScript application using Clean Architecture. The goal is to implement an SQLite `QueryBuilder` for a DAO (Data Access Object) that dynamically translates an Abstract Syntax Tree (AST) of search filters into an SQLite query using `JSON1` functions (`json_extract` and `json_each`).

We also need comprehensive Unit Tests and Integration Tests (using an in-memory SQLite database) to verify the SQL generation and execution.

## 1. The Database Schema

Documents are stored in an SQLite table (e.g., `documents`). 
All dynamic metadata is stored as a stringified JSON object in a single `TEXT` column named `metadata`.
Pagination is handled by the client, so the QueryBuilder should NOT handle `LIMIT` or `OFFSET`. It only generates the `WHERE` clause and its bound parameters.

## 2. The Domain Models

The QueryBuilder will receive a pure Domain Object representing the AST. Here are the types it will work with:

```typescript
export type LogicOperator = 'AND' | 'OR';
export type SearchOperator = 'EQ' | 'GT' | 'LT' | 'LIKE' | 'IN' | 'ELEM_MATCH';

export interface SearchCondition {
  readonly path: string; // e.g., "DocumentoInformatico.Tipologia"
  readonly operator: SearchOperator;
  readonly value: any;
}

export interface SearchGroup {
  readonly logicOperator: LogicOperator;
  readonly items: ReadonlyArray<SearchCondition | SearchGroup>;
}

export class SearchDocumentsQuery {
  constructor(public readonly filter: SearchGroup) {}
}
```

## 3. Translation Rules (AST to SQLite JSON1)

The `QueryBuilder` must expose a method (e.g., `buildWhereClause(group: SearchGroup): { sql: string, params: any[] }`) that recursively parses the tree according to these rules:

**Rule A: Path Formatting**
Every `path` must be prefixed with `$.` to form a valid SQLite JSON path (e.g., `path: "Doc.Type"` becomes `'$.Doc.Type'`).

**Rule B: Standard Operators**
Translate operators to standard SQL bindings to prevent SQL injection.
* `EQ` -> `json_extract(metadata, '$.path') = ?`
* `GT` -> `json_extract(metadata, '$.path') > ?`
* `LIKE` -> `json_extract(metadata, '$.path') LIKE ?`
* `IN` -> `json_extract(metadata, '$.path') IN (?, ?, ...)`

**Rule C: Grouping (AND / OR)**
Recursively evaluate `items`, joining the resulting SQL strings with the `logicOperator` and wrapping them in parentheses `(...)`.

**Rule D: Array Iteration (ELEM_MATCH)**
When the operator is `ELEM_MATCH`, we are searching inside a JSON array.
* The `path` of the condition is the **Base Path** (the array).
* The `value` is a `SearchGroup` containing the **Relative Paths** inside the array objects.
* Generate an `EXISTS` subquery using `json_each`.
* **Template:**
  ```sql
  EXISTS (
    SELECT 1 FROM json_each(metadata, '$.<base_path>') AS el
    WHERE json_extract(el.value, '$.<relative_path_1>') = ? 
      AND json_extract(el.value, '$.<relative_path_2>') = ?
  )
  ```
